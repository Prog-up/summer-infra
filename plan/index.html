Dossier de projet · Lab éducatif · Juin 2026

Infrastructure Cloud souveraine, auto-gérée de bout en bout

Monter, opérer et casser une plateforme Kubernetes complète sur le Public Cloud d'Infomaniak — control plane compris — pilotée intégralement en Infrastructure-as-Code et GitOps. Objectif : manipuler pour apprendre, le plus large possible, sans dépasser un crédit d'essai.

Provider · Infomaniak (OpenStack, CH) Cluster · kubeadm self-managed IaC · Terraform + Ansible CD · ArgoCD (GitOps) IAM · Keycloak (OIDC) Agentique · MCP + A2A Budget · CHF 300 / 3 mois Équipe · 2 personnes
01 — CONTEXTE & OBJECTIFS

Apprendre en construisant, pas en consommant

Le projet est un laboratoire à but pédagogique mené à deux. La règle directrice : tout ce qui est instructif doit rester sous notre contrôle. On écarte donc volontairement le Kubernetes managé, qui masque le cœur de l'administration (bootstrap, etcd, PKI, apiserver), au profit d'un cluster monté à la main avec kubeadm sur des VM brutes.

L'ambition n'est pas « un cluster qui tourne », mais une infrastructure assez complète pour exercer un maximum de domaines : provisioning déclaratif, CI/CD sécurisée, GitOps, observabilité, sécurité en profondeur, scaling, et gestion des secrets de production.

Compétences visées

  • Cluster — bootstrap kubeadm, etcd (backup/restore), PKI, CNI installé soi-même, intégration cloud OpenStack.
  • IaC / GitOps — Terraform + Ansible pour l'infra, ArgoCD pour l'état du cluster, reconstruction reproductible à chaque session.
  • Sécurité — runtime eBPF, policies d'admission, chaîne supply-chain signée, et gestion des secrets centralisée.
  • Exploitation — observabilité 3 piliers, scaling horizontal pods et nœuds, sauvegarde/restauration.
02 — CONTRAINTES & RÉALITÉ BUDGÉTAIRE

Un crédit unique, un modèle éphémère

Le levier financier est le crédit d'essai du Public Cloud : CHF 300 à consommer en 3 mois (≈ 315 €, et non « 300 € »). Deux corrections de cadrage, vérifiées en amont :

  • Le programme étudiant d'Infomaniak offre de l'hébergement web et de la messagerie — aucun crédit Public Cloud. Le plan « 600 € via deux comptes étudiants » n'existe pas.
  • Empiler le crédit d'essai sur deux comptes est possible mais flirte avec les CGU (un essai par organisation). Inutile, de toute façon : voir ci-dessous.
Le pivot qui rend le projet finançable Les instances éteintes ne sont pas facturées (hors stockage, IPv4 publique et GPU). Couplé au tout-IaC, l'infra devient éphémère : terraform apply avant une session, terraform destroy après. L'état durable (state, dumps SQL, snapshots etcd) vit dans l'Object Storage S3, à quelques centimes. Coût réel estimé : ~CHF 30–35/mois, soit ~CHF 100 sur trois mois. Un seul crédit suffit pour deux personnes.

Le risque budgétaire numéro un n'est pas le dimensionnement : c'est un nœud oublié allumé un week-end. Un destroy en fin de session, scripté dans un Makefile, est la principale discipline à tenir.

03 — DÉCISIONS D'ARCHITECTURE

Pourquoi self-managed, et ce que le terrain impose

Le Public Cloud d'Infomaniak repose sur OpenStack, et le projet OpenStack sous-jacent est accessible — on peut donc piloter VM, réseaux, volumes et load balancers via le provider Terraform OpenStack, exactement comme pour un cluster monté à la main.

Trois propriétés du terrain orientent la conception :

  • CNI Cilium — qu'on installe nous-mêmes : on hérite des NetworkPolicy L3/L4 et L7, de Hubble (observabilité réseau) et de Tetragon (runtime eBPF).
  • Octavia comme load balancer — exposé via le composant cloud-provider-openstack (OCCM) qu'on câble manuellement, tout comme Cinder CSI pour les volumes persistants.
  • Object Storage S3 — le socle de persistance qui survit aux destroy : state Terraform, snapshots etcd, backups de base, artefacts CI.

Conséquence directe : ce que le managé offrait « clé en main » (control plane gratuit, autoscaling de nœuds, intégration cloud) devient ici un exercice. C'est le but.

flux runtime provisioning (Terraform) bootstrap (Ansible/kubeadm) stockage / backup S3 opéré par toi / sécurité éphémère permanent
04 — CHAÎNE IaC & SÉQUENCE DE BOOTSTRAP

Trois outils, trois responsabilités étanches

Terraform/OpenTofu dessine l'infra physique (VM, réseau, security groups, IPv4, application credential OpenStack, buckets S3). Ansible transforme des VM nues en cluster. ArgoCD peuple le cluster depuis Git. La reconstruction complète est rejouée à chaque session — c'est la leçon, pas une corvée.

1

Terraform — l'infra

1 VM control-plane + 2 workers, réseau privé, security groups, IPv4 flottante, credential OpenStack pour Cinder/Octavia, buckets S3.

2

Ansible — le cluster

containerd + paquets kube → kubeadm init (--cloud-provider=external, pod CIDR cohérent) → installation de Cilium → kubeadm join des workers.

3

Intégration cloud

Déploiement de cloud-provider-openstack : Cinder CSI (volumes) + Octavia OCCM (services type: LoadBalancer), creds injectés en secret.

4

ArgoCD prend le relais

Bootstrap d'ArgoCD, puis l'app-of-apps déploie tout le reste depuis le repo GitOps. Plus rien ne se fait à la main dans le cluster.

Responsabilités d'exploitation nouvelles etcd est désormais ton problème : etcdctl snapshot save vers S3 par cron, et un restore répété au moins une fois. Un type: LoadBalancer = un LB Octavia facturé → un seul LB d'ingress, tout le reste en ClusterIP.
05 — RÔLE DE LA CI/CD GITLAB

La CI construit et vérifie. Elle ne déploie pas.

La frontière est le principe de sécurité central : le déploiement est tiré par ArgoCD (pull-based). La CI ne reçoit jamais le kubeconfig du cluster. Son rôle s'arrête à produire un artefact signé et à mettre à jour le dépôt GitOps.

  • Build rootless (Kaniko/Buildah), image taggée par SHA de commit.
  • Vérification : lint, SAST, secret detection (gitleaks), dependency scanning.
  • Supply chain : Trivy + SBOM (Syft) + signature Cosign — ce que Kyverno verifyImages exigera côté cluster.
  • Pipeline IaC séparé : terraform plan en MR, apply derrière approbation manuelle, avec tfsec/checkov.
  • Handoff : mise à jour du tag/values dans le repo GitOps ; ArgoCD synchronise.
06 — GESTION DES SECRETS

Le fil conducteur de toute l'infra

Les secrets traversent tout : la CI a besoin des creds registry et des clés de signature, le cluster des creds OpenStack, S3 et base de données, les apps de leurs tokens. On les centralise au lieu de les éparpiller. Progression en trois temps, du plus simple au plus puissant.

  • Niveau 1 — sealed-secrets : chiffrement dans le repo, GitOps-pur, zéro infra externe. Premier réflexe.
  • Niveau 2 — SOPS + age : chiffrement des values.yaml et tfvars, déchiffrés par ArgoCD ou en CI.
  • Niveau 3 — Vault + external-secrets : le vrai sujet. Secrets dynamiques (user DB à TTL court), moteur PKI (certs mTLS internes), transit (chiffrement-as-a-service), auth Kubernetes (un pod s'authentifie via son ServiceAccount, sans secret en dur).
07 — GESTION DE L'IDENTITÉ (IAM)

Keycloak, point d'autorité unique

Après les secrets, l'identité est le second sujet transverse : un seul fournisseur, Keycloak, gouverne à la fois l'accès humain (SSO sur les outils) et l'accès au cluster lui-même. C'est un bénéfice direct du self-managed — on configure l'authentification OIDC dans l'apiserver, ce qu'un control plane en boîte noire ne permet généralement pas.

  • SSO plateforme — Grafana, ArgoCD, Hubble UI derrière un seul login OIDC. Fini les comptes locaux par outil.
  • Accès au cluster (le gros gain self-managed) — kube-apiserver configuré avec Keycloak comme issuer OIDC ; kubectl via le plugin kubelogin obtient des jetons courts. Les groupes Keycloak sont mappés sur le RBAC Kubernetes — fini les certs statiques dans le kubeconfig.
  • Apps non-OIDC — oauth2-proxy en forward-auth (annotation ingress-nginx) place n'importe quel service derrière Keycloak sans le modifier.
  • Realm-as-code — realm, clients et groupes versionnés via terraform-provider-keycloak ou keycloak-config-cli, recréés à chaque apply : cohérent avec le modèle éphémère.
  • Dépendances bouclées — Keycloak s'adosse à CloudNativePG pour son état, et ses secrets (admin, client secrets) sont servis par Vault/external-secrets. Identité et secrets se renforcent.
Pourquoi Keycloak ici, et non un service géré L'intérêt pédagogique est d'opérer le fournisseur et de câbler l'OIDC de l'apiserver à la main — les drapeaux --oidc-issuer-url et la map groupes→RBAC sont à toi. Keycloak étant gourmand, il démarre comme module « Identité » puis peut être promu vers le core dès que les nœuds le permettent.
08 — SÉCURITÉ EN PROFONDEUR

Défense par couches, du réseau au runtime

  • Réseau zero-trust — default-deny par namespace, puis CiliumNetworkPolicy (filtrage L7 inclus). Hubble UI pour écrire les policies à partir du trafic réel observé.
  • Runtime eBPF — Tetragon (natif Cilium, observe et applique) en premier ; Falco en complément pour la richesse de règles. Sonde eBPF moderne, pas de module kernel.
  • Admission policy-as-code — Kyverno : interdiction du privileged/root, requests/limits obligatoires, registries restreints, et verifyImages (signatures Cosign).
  • Pod Security Admission — mode restricted sur les namespaces applicatifs.
  • Supply chain bouclée — trivy-operator scanne en continu les workloads qui tournent ; rien ne s'exécute sans avoir été scanné et signé en CI.
  • Posture — kube-bench (CIS), durcissement OS des VM via Ansible.
Limite assumée L'accès à la politique d'audit de l'apiserver dépend du control plane — mais ici tu opères le control plane, donc l'audit log et l'etcd sont entièrement à ta main. C'est précisément l'avantage du self-managed.
09 — OBSERVABILITÉ & SCALING

Voir, puis encaisser la charge

Observabilité 3 piliers

kube-prometheus-stack (métriques + Grafana + Alertmanager), Loki (logs), Tempo + OpenTelemetry Collector (traces). Alertmanager route vers mail/kChat. Les logs Falco et l'audit partent dans Loki pour un SIEM léger.

Scaling horizontal — trois niveaux

  • Pods — HPA (CPU/RAM via metrics-server), puis KEDA pour l'event-driven (scale sur une query Prometheus ou une file, jusqu'au scale-to-zero).
  • Nœuds, manuel/IaC — bump du compte de workers en Terraform → kubeadm join. On voit le nœud rejoindre : plus instructif qu'un autoscaler opaque pour débuter.
  • Nœuds, automatique — Cluster API (CAPI) + cluster-autoscaler, l'étape avancée une fois le bootstrap maîtrisé.

Résilience pendant le scale : PodDisruptionBudget, topologySpreadConstraints multi-zone, anti-affinity.

10 — APPROCHE « CORE + MODULES »

Tout n'a pas à tourner en même temps

Deux petits workers ne peuvent pas héberger simultanément Vault, Keycloak, le mesh, l'observabilité complète et les apps. On distingue un socle léger toujours déployé et des modules thématiques qu'on allume à la séance (en montant les nœuds le temps voulu). Chaque module est un dossier du repo GitOps activé par un flag — ce qui est, en soi, un exercice GitOps.

event-driven

Broker + worker

Uploads vers l'Object Storage → exerce les creds S3 via Vault/external-secrets.

Sauvegardes, rapports planifiés → primitives Job/CronJob.

Status page

SSO Keycloak

11 — COUCHE AGENTIQUE (MCP & A2A SÉCURISÉS)

Faire tourner des agents sans ouvrir de brèche

C'est là que la plateforme paie. MCP (serveurs d'outils pour LLM) et A2A (communication agent-à-agent) ont tous deux été pensés pour la communication, pas pour la sécurité : ils délèguent l'authentification à l'implémenteur. Nos briques existantes — Keycloak, Vault, le maillage Cilium, la stack runtime — sont précisément ce qui ferme ces trous. On ne réinvente rien, on réutilise.

État courant (vérifié, mi-2026) : les serveurs MCP sont désormais des Resource Servers OAuth 2.1, qui publient leur Authorization Server via Protected Resource Metadata (RFC 9728) et exigent côté client les Resource Indicators (RFC 8707). A2A, lui, traite les agents comme des applications d'entreprise, l'identité vivant au niveau du transport HTTP, jamais dans la charge utile ni dans l'Agent Card.

  • Une seule autorité — Keycloak (§07) sert d'Authorization Server OAuth 2.1/OIDC pour MCP (serveurs = Resource Servers, découverte RFC 9728) et pour A2A (agents authentifiés comme apps, JWT). Humains et agents partagent le même IdP.
  • Jetons courts + rotation — le cœur MCP ne mandate pas le cycle de vie des bearer tokens (gap signalé par la note de sécurité NSA de mai 2026 : risque de rejeu/hijack). Vault émet des jetons à TTL court et scoping serré ; RFC 8707 côté client évite le détournement de token entre serveurs.
  • A2A « sécurisé » — le protocole ne vérifie pas l'authenticité des Agent Cards. On ajoute mTLS (Cilium mesh, sans sidecar), Agent Cards signées (PKI Vault), validation du contenu des cards (anti context-poisoning), et creds toujours hors de la card.
  • Confinement réseau — default-deny egress + allowlist L7 Cilium par namespace d'agent : un MCP server ou un agent ne joint que les endpoints autorisés (anti-exfiltration, anti-SSRF), via une egress gateway.
  • Runtime & supply chain — Tetragon/Falco surveillent les exec (un agent exécute des appels d'outils et du code généré), Kyverno + PSA restricted confinent les workloads, et les images de serveurs MCP sont scannées et signées comme le reste.
  • Scaling — le cœur MCP devient stateless (RC 2026) : scale horizontal derrière l'ingress avec HPA/KEDA. Le tracing Tempo/OTel suit les chaînes d'appels d'agents.
À ne pas survendre L'infra ferme les trous d'authentification et de confinement. Elle ne neutralise pas l'injection de prompt via le contenu d'un outil ou d'une donnée — le risque le plus dur, en partie hors périmètre infra. On le réduit par des scopes au plus juste, un humain dans la boucle pour les actions sensibles, et l'allowlist d'egress. Aucune brique ne dispense de cette prudence.
12 — BUDGET & DISCIPLINE

Estimation et garde-fous

  • Socle permanent — VM GitLab (~CHF 16/mois) + Block Storage + IPv4 + S3 ≈ CHF 20–25/mois, incompressible.
  • Cluster éphémère — control plane + workers facturés uniquement allumés ; ~4 h × ~12 séances ≈ CHF 5–10/mois.
  • Total ≈ CHF 30–35/mois → ~CHF 100 sur 3 mois. Un seul crédit CHF 300 couvre les deux personnes.
Garde-fou n°1 terraform destroy en fin de séance, scripté dans un Makefile et accompagné d'un rappel. C'est là que se jouent 90 % des dépassements de crédit, jamais dans le dimensionnement.
